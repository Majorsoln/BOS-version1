Six-phase lifecycle: template selection → content assembly → **compliance binding** (pack version at issuance, D-009) → numbering → hash → issuance (`document.issued.v1`). **Two-layer integrity:** the document hash (over content + template-version ref + pack-version ref + number) is stored **inside the issuance event**, so it is itself chain-protected (CN-4-003) — the document hash enables standalone/offline verification, the event chain protects it in-store. **Integrity vs authenticity:** the hash proves the content is unchanged (integrity) but not that BOS issued it (authenticity) — offline authenticity may need a **cryptographic signature**; flagged to Term 7 via CTR-017. **Numbering** is tenant-isolated, branch/terminal-scoped, sequential, deterministic, **block-reserved for DEGRADED** (D-007 #12); reconciliation is **compliance-pack-driven** (gap-free jurisdictions continue the sequence; gap-tolerant mark reserved-but-unused with an auditable reason). The hash covers **content + version references, not rendered output** — re-display uses the referenced template version. **Template versioning + freeze** (old documents never re-rendered; old templates retained). **Corrections/cancellations = new documents** referencing the original (D-009). **CTR-017 → NEGOTIATING** (Foundation verification logic delivered; integrity-vs-authenticity/signature question raised for Term 7). No new CTRs; no Charter conflict.

- **Term 4 CN-4-009 (Replay Engine)** ratified and merged (commit `0d80051`). The mechanics of reconstructing state from events (proof of CN-4-001 Assertion 6). Four replay types: full / scoped (tenant, aggregate) / point-in-time / projection rebuild. **Determinism** via five requirements — same events, same `global_position` order, same **rules-data** (D-009 pack-version freeze), same **rules-code** (fold/handler code stays stable through additive versioning; changes become new event-type versions with **version-aware handlers**, CN-4-010 — determinism through version-aware handlers, *not* "code freeze"), same time (CN-4-014). **Defensible/audit replay includes hash-chain verification** (CN-4-003) for the scope; output states whether the chain was intact (routine rebuilds skip verification for speed). **Point-in-time cutoff is `global_position`** (a timestamp request resolves once to the position of the last event at/before it). **Checkpoints** are non-truth (CN-4-018), verified at creation, and **invalidated by a hash breach** at/before their position. **No-skip / compensating-events-only** correction (O9). **Replay never emits events** — pure reconstruction; persisting a rebuilt projection is a derived write, not event emission (may be deferred in READ_ONLY). No new CTRs; no Charter conflict.

Formalises read-models that several merged docs already rely on (audit log CN-4-008, Decision Journal CN-4-013, engine registry CN-4-020). Projections are derived, rebuildable, deterministic, disposable, queryable, **position-aware**. **Position tracking & freshness:** each projection exposes `processed_position` / `processed_timestamp` — the **source of the Advisor Framework's `freshness_indicator`** (CN-4-022 N4). Tenant-scoped enforcement by the framework, not handlers. No new CTRs; no Charter conflict.
